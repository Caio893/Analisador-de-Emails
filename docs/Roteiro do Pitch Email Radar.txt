Roteiro do Pitch: Email Radar

1. O Gancho e o Problema (0:00 - 0:45)
O que falar: > "Vocês sabiam que mais de 80% dos ataques cibernéticos bem-sucedidos hoje começam com um simples e-mail de phishing? Diariamente, pessoas físicas e colaboradores de empresas clicam em links maliciosos porque os golpes estão cada vez mais sofisticados, utilizando IA para clonar a identidade visual e a linguagem de marcas reais. Os filtros tradicionais de spam deixam passar muitas dessas ameaças e, quando bloqueiam, não explicam ao usuário por que aquilo é perigoso, deixando-o vulnerável no próximo e-mail."

O que mostrar no slide: Uma imagem limpa ou dados estatísticos rápidos sobre Phishing e Engenharia Social.

2. A Solução (0:45 - 1:30)
O que falar:

"Para resolver isso, eu desenvolvi o Email Radar (ou MailGuard AI). Uma plataforma full-stack de segurança acoplada ao Gmail que funciona como uma camada inteligente de proteção. O usuário se conecta de forma simples via Google OAuth e o nosso sistema sincroniza as mensagens. A mágica acontece em seguida: nossa IA analisa o conteúdo e classifica o risco do e-mail em quatro níveis claros: de confiável a altamente perigoso. Mais do que isso, a plataforma gera uma explicação didática e transparente em português, alertando o usuário sobre inconsistências antes que ele clique em algo perigoso."

O que mostrar no slide: Print do seu Dashboard ou da caixa de entrada do app (/app/inbox) mostrando o e-mail classificado com a caixinha de explicação da IA.

3. Engenharia e Arquitetura: Os Diferenciais Técnicos (1:30 - 2:30)
(Esta é a parte onde você ganha nota máxima com os professores de tecnologia/computação)

O que falar:

"Como engenheiro/desenvolvedor do projeto, a minha principal preocupação não foi apenas fazer a IA funcionar, mas sim a arquitetura e a segurança. Desenvolvi o projeto utilizando Django REST Framework no backend e React com TypeScript no frontend.
E aqui estão os três grandes destaques de engenharia do projeto:

Privacidade Rigorosa: O frontend nunca tem acesso a chaves da OpenAI ou do Google. Além disso, os tokens OAuth dos usuários são criptografados em repouso no banco PostgreSQL usando criptografia Fernet. O payload enviado para a IA é minimizado, removendo URLs brutas para proteger os dados.

IA Resiliente (Fallback Local): Se a API da OpenAI falhar ou demorar, a aplicação não quebra. Implementei heurísticas locais no backend que assumem o controle e garantem uma classificação preliminar para o usuário.

Pronto para Produção: Toda a stack é conteinerizada com Docker. O tráfego passa por um proxy reverso com Caddy, garantindo HTTPS automatizado, integrado a uma fila assíncrona com Redis para o processamento de e-mails de forma otimizada."

O que mostrar no slide: O diagrama de blocos da sua arquitetura (Frontend <-> Backend <-> Redis/PostgreSQL <-> Gmail/OpenAI).

4. Demonstração / Validação (2:30 - 3:00)
O que falar:

"O sistema já está totalmente funcional e publicado no domínio email-radar.com. Para fins de testes e auditorias de interface, a aplicação também conta com um modo mock isolado, permitindo que qualquer avaliador navegue pelas funcionalidades, busque remetentes e veja as análises de risco simuladas em tempo real sem precisar vincular uma conta real logo de início."

O que mostrar no slide: Um gif rápido da tela funcionando ou um QR Code direcionando para o site ou repositório.

5. Conclusão e Próximos Passos (3:00 - 3:30)
O que falar:

"O Email Radar prova que é possível unir Inteligência Artificial avançada, privacidade de dados e uma interface moderna orientada ao produto para mitigar o elo mais fraco da segurança digital: o fator humano. Como próximos passos do Roadmap, planejo expandir o sistema para políticas multi-usuário e times corporativos. Muito obrigado, e estou aberto às perguntas da banca."

💡 Dicas Estratégicas para as Perguntas da Banca
Como o seu projeto mexe com e-mail e IA, os professores tendem a fazer perguntas previsíveis. Já se prepare para responder:

"E se o usuário estiver preocupado em dar acesso ao e-mail dele?"

Resposta na ponta da língua: "Nós mitigamos isso limitando o escopo do Google estritamente para gmail.readonly. O sistema não tem permissão para apagar, enviar ou modificar nada. Além disso, as chaves são criptografadas no banco e o usuário pode revogar o token e apagar seus dados locais da plataforma com um clique."

"O custo da API da OpenAI não ficaria muito alto se o usuário recebesse muitos e-mails?"

Resposta na ponta da língua: "Pensando nisso, o sistema faz uma redução prévia do payload (reduzindo o texto e limpando ruídos) para gastar o mínimo de tokens possível na OpenAI. Além disso, no roadmap, o uso de filas com Redis nos permite criar políticas de priorização e, no futuro, podemos adotar modelos locais de código aberto (open-source) rodando no próprio servidor."
